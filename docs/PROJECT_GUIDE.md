# プロジェクトおよびサンドボックス払い出しガイド

本ドキュメントは、Google Cloud プロジェクトの新規作成、サンドボックスの払い出し、およびデプロイの手順について記述したものです。

---

## 1. プロジェクトの申請と作成

### 1.1 通常プロジェクト (App Projects)
1.  `governance/admin/factory/inventory.json` を更新します。
2.  PR を作成し、マージされると自動的にプロジェクトが作成されます。

### 1.2 サンドボックス (Sandboxes)
1.  GitHub Actions の **「GCP: Create Sandbox」** ワークフローを実行します。
2.  `sandbox_id` と `github_repo` を入力して実行すると、自動的に `inventory.json` が更新され、プロジェクトが払い出されます。

---

## 2. セットアップ手順 (重要)

プロジェクト作成が完了すると、基盤ボットが**自動的にあなたのリポジトリの初期設定を行います。**

### 2.1 GitHub Variables の自動同期
1.  基盤側の処理が完了すると、あなたのアプリリポジトリの **「Settings」>「Secrets and variables」>「Actions」** に、デプロイに必要な ID 群（`ID_STG`, `ID_PRD`, `APP_NAME` 等）が自動的に書き込まれます。
2.  開発者は Slack 通知、または GitHub の設定画面を確認し、設定が届いていることを確認するだけです。**YAML への手動コピペは不要です。**

### 2.2 本番環境 (prd) の承認ルール設定
本番環境への誤デプロイを防ぐため、以下の設定を GitHub リポジトリで行うことを強く推奨します。

1.  GitHub リポジトリの **「Settings」>「Environments」** を開きます。
2.  **「New environment」** ボタンを押し、**`prd`** という名前で作成します。
3.  **「Required reviewers」** にチェックを入れ、デプロイ承認者を指定して保存します。
    - これにより、本番デプロイ時に指定された人の承認ボタンが必要になります。

---

## 3. デプロイの実行

本基盤の雛形ワークフローは、GitHub Variables を自動的に読み込み、**「インフラ（Terraform）」と「アプリプログラム（gcloud等）」** を適切な順序でデプロイします。

### 3.1 通常デプロイ (`app_deploy.yml`)
本基盤の雛形は、安全かつ効率的な **「多段階スマート・パイプライン」** を採用しています。

#### デプロイの流れ
1.  **DEV環境 (自動)**: `main` ブランチへのプッシュで自動実行されます。
    - ※ Variables に `ID_DEV` が設定されていない場合は、自動的に **Skipped (灰色 ⚪️)** となり、次のステップへ進みます。
2.  **STG環境 (自動)**: DEVデプロイが成功（またはスキップ）した後に自動実行されます。
3.  **PRD環境 (手動)**: 本番事故防止のため、自動実行はされません。GitHub Actions の画面から手動で実行（workflow_dispatch）する必要があります。

#### 特徴
- **正直なステータス**: デプロイが実行されたのか、環境未設定で飛ばされたのかが GitHub 上で明確に区別されます。
- **連鎖の停止**: 前の環境でエラーが発生した場合、後続の環境へのデプロイは安全のために自動的にキャンセルされます。

### 3.2 サンドボックスデプロイ (`sandbox-deploy.yml`)
- `sandbox/**` ブランチへのプッシュ、または手動実行。
- 自分のサンドボックスプロジェクト内だけで、自由に実験が可能です。

---

## 4. ライフサイクルと自動削除

- **サンドボックスの期限**: 作成時に指定した期限（デフォルト 7日）が過ぎると自動削除されます。
- **削除フロー**: ボットが `inventory.json` から定義を削除し、Terraform が物理削除を行います。
- **ゾンビ防止**: 台帳（Git）と実体（GCP）が常に同期されるため、削除後の不要な課金は発生しません。
