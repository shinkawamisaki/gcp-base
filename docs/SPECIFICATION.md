# GCP-base: 機能詳細仕様書 (Technical Specifications)

本ドキュメントは、GCP-base が提供する主要機能の動作メカニズム、オートメーションのシーケンス、および技術的な制約について定義したものです。

---

## 1. プロジェクト・ファクトリー (Project Factory)

`inventory.json` を唯一の正本（Source of Truth）として、GCP プロジェクトとその周辺リソースを自動一括生成する仕組みです。

### 1.1 動作シーケンス
1.  **定義更新**: 管理者が `governance/admin/factory/inventory.json` にアプリ名、環境、GitHub リポジトリを追記。
2.  **ID 生成**: `random_id` モジュールにより、ユニークなサフィックス（例: `-f3a1`）を自動付与。
3.  **プロジェクト作成**: 指定されたフォルダ配下にプロジェクトを作成。
4.  **API 有効化**: 基本 API を一括有効化。
5.  **権限・WIF 連携**: 
    - **動的 WIF バインディング**: `inventory.json` に記載された `github_repo` に完全一致するリポジトリのみが、そのプロジェクトの SA に成り代われるように制限。
    - **環境別権限**: `stg/audit` は Editor, `prd` は Viewer を付与。
6.  **予算アラート**: プロジェクト単位の予算上限設定と、Pub/Sub への通知紐付け。

### 1.2 堅牢性のための実装
- **待機タイマー**: GCP 内部の非同期なプロビジョニングによるエラーを防ぐため、120s~180s の待機を挿入。
- **削除防止ロック (Lien)**: `is_sandbox = false` のプロジェクトには誤削除防止ロックを自動付与。

## 2. サンドボックス・ライフサイクル (Sandbox Lifecycle)

有効期限が来ると「台帳（Git）」と「実体（GCP）」の両面からクリーンに自動削除される、ゾンビ化防止型の仕組みです。

### 2.1 台帳同期型・自動削除フロー (GitOps Cleanup)
ボットによる物理削除の直接実行を廃止し、以下の 4 ステップで削除を完遂します。

1.  **検知**: `sandbox_lifecycle` ボットが 1時間おきに稼働。ラベル `expiry_date` を現在時刻と比較。
2.  **削除依頼 (Slack & GitHub)**: 期限切れを検知すると、ボットが Slack に「自動クリーンアップ開始」を通知し、GitHub API を叩いて `platform-delete-sandbox.yml` を起動。
3.  **台帳更新**: ワークフローが `inventory.json` から当該サンドボックスの定義を自動削除し、`main` に Push。
4.  **物理削除**: 台帳更新をトリガーに Terraform が起動。「定義が消えた」ことを検知し、GCP 上のプロジェクトを安全に物理削除（destroy）する。

## 3. アプリケーション・デプロイ戦略 (Handover Strategy)

職務分掌を維持しつつ、本番環境へのデプロイを実現する「特権委譲」の仕組みと、環境を跨ぐパイプラインの仕様です。

### 3.1 デプロイメント・アイデンティティ
- **検証/開発環境**: 各プロジェクトの `manager-sa`（Editor）を使用。
- **本番環境**: プロジェクト SA は Viewer 権限のみ。デプロイ時は管理プロジェクトの `runner-sa` に権限を委譲（Handover）して実行。
- **サンドボックス**: `dev-manager-sa` を使用し、自身のプロジェクト内のみを操作。

### 3.2 多段階スマート・パイプライン (Smart Pipeline)
アプリリポジトリの `app_deploy.yml` は、以下の論理構造で動作します。

1.  **環境自動スキップ (Conditional Skip)**: 
    GitHub Variables (`ID_DEV` 等) の設定有無を `if` 条件で判定。未設定の環境はジョブ自体を **Skipped (灰色 ⚪️)** とし、画面上のステータスの誠実さを担保します。
2.  **逐次実行 (Sequential Chaining)**:
    `DEVデプロイ` → `STGデプロイ` → `PRDデプロイ` の順に依存関係 (`needs`) を構築。前の環境が失敗した場合、後続環境へのデプロイは自動的にキャンセルされます。
3.  **保護環境の分離**:
    `environment` 設定により、`prd` 環境のみ GitHub 上での手動承認（Manual Approve）を必須とする制御が可能です。

## 4. 通知と自動セットアップ (Automation & DevX)

### 4.1 GitHub Variables 自動同期 (Secretless Config)
プロジェクトの作成・更新完了後、基盤側の Python ボット (`notify_inventory_changes.py`) が、対象のアプリリポジトリに対して以下の設定を自動で流し込みます。これにより、開発者による手動のコピペ作業を「0」にします。

- **同期対象リソース**: 対象リポジトリの `Settings > Secrets and variables > Actions > Variables`
- **同期される値**:
    - `APP_NAME`: アプリの識別名
    - `ADMIN_PROJECT_ID / NUMBER`: 管理基盤の情報
    - `ID_DEV / ID_STG / ID_PRD`: 各環境の GCP プロジェクト ID
- **動作保証**: 1つのリポジトリの同期に失敗しても他の処理を止めない「非ブロッキング同期」を採用。失敗時は Slack にエラーレポートを配信します。

### 4.2 スマート通知ロジック
- `inventory.json` の Git 差分を自動解析。
- **「設定変更があったプロジェクト」** だけを特定して通知し、不要なスパムを防止します。
- **通知チャンネルの分離**: インフラ管理者向けと開発者向けの通知を、Webhook の切り替えにより適切に分離配信します。

## 5. 職務分掌と権限隔離 (Separation of Duties)

IPO 審査における「内部統制」の要件を満たすため、強力な権限が 1 つの SA に集中しないように物理隔離しています。

- **特権 SA (`prd-org-policy-sa`)**: 組織ポリシーのみを変更可能。実プロジェクトの作成は不可。
- **通常 SA (`prd-terraform-runner-sa`)**: プロジェクト作成やリソース操作を担当。組織ポリシーの変更は不可。
- **Manager SA**: 各アプリプロジェクト内の操作のみを担当。他プロジェクトや管理基盤へのアクセスは不可。
- **WIF 制限**: 各リポジトリ（GitHub）は、自分に関連する Manager SA にしか成り代われないように制限。

---

## 6. プロジェクト削除の統制仕様 (Governance of Project Destruction)

不注意や不正による重要プロジェクトの消去を物理的に防ぐための仕様です。

### 6.1 削除保護メカニズム (`PREVENT`)
- **デフォルト保護**: 通常のアプリプロジェクトには `deletion_policy = PREVENT` が設定されています。台帳から定義を削除しただけでは、Terraform は削除を拒否しエラーを返します。
- **特権実行**: 物理削除を完遂するには、管理者が内容を精査した上で、明示的に `-var="deletion_policy=DELETE"` フラグを付与して手動実行する必要があります。
- **二重のロック (Lien)**: Terraform の保護に加え、GCP リソースレベルでの Lien（削除防止ロック）により、コンソールからの誤操作も遮断されます。
