# GCP-base: 詳細アーキテクチャ設計ガイド

## 1. 職務分掌に基づく物理・論理隔離
本基盤はIPOを見据えた内部統制を支えるため、権限とデータをディレクトリ・Stateレベルで完全に分離しています。

- **Governance領域 (`governance/org-policies`)**: 
    - 組織全体のガードレール（憲法）を定義。
    - **特権 SA (`prd-org-policy-sa`)** が管理。
    - 台帳 (State) は `terraform/org-policies/` に隔離。
- **Admin領域 (`governance/admin`)**: 
    - 基盤プロジェクト、WIF、予算通知等の実務リソースを構築。
    - **通常 SA (`prd-terraform-runner-sa`)** が管理。
    - 台帳 (State) は `terraform/admin/state` で管理。

## 2. 最小権限マトリクス 

### 2.1 特権 SA
組織全体のセキュリティ設定を管理する、最も高い権限を持つ自動化主体です。
| 対象範囲 | ロール | 理由・目的 |
| :--- | :--- | :--- |
| 組織レベル | `roles/orgpolicy.policyAdmin` | マネージド制約を含む組織ポリシーの管理 |
| 組織レベル | `roles/resourcemanager.organizationViewer` | 組織構造の参照 |
| Adminバケット | `roles/storage.objectAdmin` (条件付) | 自身の State (Prefix) のみの読み書き |

### 2.2 通常 SA
日々のインフラ構築を担当する実務主体です。組織ポリシーを変更する権限は持ちません。
| 対象範囲 | ロール | 理由・目的 |
| :--- | :--- | :--- |
| 組織レベル | `roles/resourcemanager.projectCreator` | 新規プロジェクトの払い出し |
| 請求レベル | `roles/billing.user` | 予算・請求の紐付け |
| 請求レベル | `roles/billing.costsManager` | 予算アラートの設定・管理 (Foundation レイヤーで使用) |
| フォルダレベル | `roles/resourcemanager.folderAdmin` | Workloads/Sandboxes フォルダ内の構造管理 |
| フォルダレベル | `roles/compute.admin` | VPC・Firewall 等のネットワーク構築 |
| フォルダレベル | `roles/storage.admin` | GCS バケットの管理 |
| 管理 PJ | `roles/resourcemanager.projectIamAdmin` | 管理プロジェクト内の IAM 管理 |
| 管理 PJ | `roles/editor` | 管理プロジェクト内のリソース作成 (Cloud Functions 等のデプロイ実務のため) |

> **注記:** Runner SA (`prd-terraform-runner-sa`) の組織・フォルダレベル権限は、`bootstrap.sh` による初期セットアップ時に一度だけ付与されます。これらは、日々の `governance/admin` 配下の Terraform コードでは管理・変更の対象としません。これにより、**「強力な特権を日々の運用コードから物理的に隔離する」**という職務分掌を担保しています。

### 2.3 アプリプロジェクト用 SA
各アプリプロジェクトごとに作成される、環境別のデプロイ主体です。
| 環境 | ロール | 理由・目的 |
| :--- | :--- | :--- |
| `stg` / `audit` | `roles/editor` | 検証・監査環境での迅速なリソース構築・変更 |
| `prd` | `roles/viewer` | 本番環境での誤操作防止（※デプロイは管理 SA への委譲により実行） |

### 2.4 特権委譲デプロイ
本番環境（prd）へのデプロイには、プロジェクト固有の SA ではなく、管理プロジェクトの `Runner SA` を使用します。これにより、本番環境への変更操作を中央で厳格に監査・制御する「Handover 戦略」を実現しています。

### 2.5 動的 WIF バインディング
`inventory.json` に記載された GitHub リポジトリ名に基づき、リポジトリ単位で SA の使用許可を動的に制限します。ワイルドカードを排した「1リポジトリ 1SA」の原則を自動で構成します。

### 2.6 ビルド・デプロイ権限 
サーバーレス関数やコンテナのデプロイを成功させるため、管理プロジェクトの Cloud Build サービスエージェントに対して、各アプリプロジェクトへの以下の最小権限を自動付与しています。
| ロール | 理由・目的 |
| :--- | :--- |
| `roles/logging.logWriter` | ビルドログの書き込み |
| `roles/artifactregistry.admin` | イメージのプッシュ・管理 |
| `roles/cloudfunctions.admin` | Cloud Functions (Gen2) のデプロイ |
| `roles/run.admin` | Cloud Run サービスの管理 |

## 3. ガードレール設計
本基盤では、レガシーなドメイン制限 (`iam.allowedPolicyMemberDomains`) を無効化（Allow All）し、より詳細かつ厳格な制御が可能な **マネージド制約** に一本化しています。

- **`iam.managed.allowedPolicyMembers`**: 
    - 組織メンバーおよび、特定の Google 公式サービスアカウント（予算通知等）のみを例外として許可。
    - `enforce = TRUE` により、意図しないドメインの SA 追加を物理的に遮断します。

## 4. 堅牢性と Security by Design

### 4.1 シークレットの最小権限アクセス (Secret Manager)
Runner SA が管理プロジェクト内のすべてのシークレットを読み取れるわけではありません。IAM Condition を活用し、特定の接頭辞を持つシークレットのみにアクセスを制限しています。
- **許可対象**: `prd-` または `infra-` で始まるシークレット名。
- **目的**: 開発環境や他システムの機密情報への不要なアクセスを遮断する、多層防御（Defense in Depth）の実装。

### 4.2 誤削除防止ロック
Project Factory を通じて作成された本番（prd）および検証（stg）プロジェクトには、自動的に **Lien (削除防止ロック)** が付与されます。
- **機能**: Terraform やコンソールからの誤った「プロジェクト削除」操作を物理的に拒否します。
- **解除**: 削除が必要な場合は、明示的にロックを解除するステップを挟むことで、人為的ミスによる大規模障害を防止します。

## 5. リアルタイム予算統制
- **グローバル SA 利用**: 予算通知には実在するグローバル SA (`billing-budget-alert@system.gserviceaccount.com`) を使用し、ドメイン制限下でも確実な通知を実現。
- **反映待ちの実装**: GCP 内部の非同期なプロビジョニングを考慮し、各種リソース間に `time_sleep` (120s~180s) を挟むことで、構築の再現性を担保しています。

## 6. ログ集約と AI 監査
本基盤には、Gemini 2.5-flash を活用した自律型のセキュリティ監査ボット (`weekly_check`) が組み込まれています。
- **フォルダ横断スキャン**: 特定のプロジェクトだけでなく、指定したフォルダ（Workloads, Sandboxes 等）配下の全プロジェクトを自動で列挙し、ファイアウォール全開放や公開バケットなどのリスクを検知。
- **高度なリスク検知**: IAP (Identity-Aware Proxy) を介さない SSH 直接露出の検知や、重要サービスの監査ログ設定状況を精査。
- **AIによる客観的な要約**: 検出された大量のログから「今すぐ対応すべき Top5」を Gemini が選別し、Slack に客観的なトーンで通知。
- **証跡保存**: 監査結果は GCS バケットに Markdown 形式で保存され、後からの監査対応（IPO 審査等）にも利用可能。

## 7. ガバナンスの強制
個別のプロジェクト設定に依存せず、組織全体で一律のガードレールを適用しています。
- **データアクセスログの一括有効化**: 組織レベルで IAM, Secret Manager, Storage の監査ログを強制有効化し、IPO 審査に耐えうるトレーサビリティを確保。
- **最小権限のビルド環境**: 監査ボットや予算通知ボット等、すべての運用系 Cloud Functions のデプロイにおいてデフォルトの強力なサービスアカウントを排除し、特定リポジトリに限定した専用ビルド SA を採用。

## 8. ネットワーク設計思想
本プロジェクトでは、単一の共有 VPC (Shared VPC) による中央集権的な管理ではなく、**「プロジェクトごとに独立したカスタム VPC」** を構築する分散型のネットワーク構成を推奨・実装しています。

- **被害半径の最小化**:
  プロジェクト単位で VPC を物理的に分離することで、特定プロジェクトにおける設定ミスや脆弱性が他のプロジェクト（特に PII を扱うプロジェクト）へ波及するリスク（横移動：Lateral Movement）を構造的に排除しています。
- **ガバナンスの自動化とセルフサービス**:
  標準化された Terraform モジュール (`vpc_base`) を通じて、安全性が担保されたネットワーク環境をオンデマンドで払い出すことが可能です。これにより、中央のインフラチームを作業ボトルネックにすることなく、統制とスピードを両立します。
- **監査の透明性**:
  プロダクト間の通信路がデフォルトで存在しない「疎結合」な状態を維持することで、内部統制におけるネットワーク境界の説明を容易にし、高い監査耐性を確保しています。

## 9. 運用管理の分離
本プロジェクトでは、運用管理の効率化とリスク低減のため、Admin 領域を **「Foundation（基盤）」** と **「Factory（工場）」** の 2 つの State に物理分離しています。

- **リスクの隔離 (Risk Isolation)**:
  WIF や権限管理などの「めったに変更しないが、壊れると全デプロイが止まる重要リソース (Foundation)」と、プロジェクトの追加・削除といった「頻繁に変更されるリソース (Factory)」を分離しています。これにより、日々のプロジェクト払い出し作業が重要基盤の設定を誤って損なうリスクを物理的に排除しています。
- **デプロイの高速化とスケーラビリティ**:
  プロジェクト数が増加しても、Factory 側の State のみをリフレッシュ・更新することで、デプロイ時間を短縮し、開発スピードを維持します。
- **職務分掌の強化**:
  将来的に「基盤を管理するチーム」と「プロジェクトの発行を承認するチーム」を明確に分けるための構造的土台を提供します。

## 10. ワークフロー構成

運用負荷の軽減と開発者体験 (DX) 向上のため、GitHub Actions ワークフローを「基盤管理用」と「アプリ雛形用」に物理分離しています。

- **基盤管理ワークフロー (Platform Tier)**:
  `.github/workflows/platform-*.yml` として集約。インフラチームが基盤全体の統制、プロジェクトの払い出し、組織ポリシーの適用に使用します。
## 11. 通知アーキテクチャとチャンネル設計 (Notification Strategy)

本基盤では、情報の重要度と対象読者に応じて Slack 通知を以下の 4 つのチャンネルに分離することを推奨しています。これらは Secret Manager 内の各シークレットに Webhook URL を設定することで有効化されます。

| シークレット名 | 通知先（推奨） | 内容・役割 |
| :--- | :--- | :--- |
| `infra-deploy-slack-webhook` | `#infra-ops` | **基盤運用**: 基盤全体の更新完了サマリー、デプロイエラー、クリティカルなシステム警告。 |
| `infra-sandbox-slack-webhook` | `#dev-sandbox-info` | **開発者支援**: サンドボックスやアプリプロジェクトの払い出し完了通知、コピペ用設定ガイドの配信。 |
| `infra-audit-slack-webhook` | `#security-audit` | **AI監査**: Gemini 2.5-flash による週次セキュリティレポート。設定ミスや高リスク項目の要約通知。 |
| `infra-billing-slack-webhook` | `#billing-alerts` | **コスト管理**: 予算の 50%, 90%, 100% 到達、および予測超過時の即時アラート。 |

> **注記**: 初期構築時など、全ての情報を一箇所で把握したい場合は、同一の Webhook URL を全てのシークレットに設定することも可能です。
